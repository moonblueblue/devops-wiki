# 카오스 엔지니어링

## 1. 개념

```
"시스템이 실제로 장애 상황에서
 얼마나 잘 동작하는지 실험으로 검증한다."

기존 방식:
  장애 발생 → 대응 → 수정
  (사후 대응)

카오스 엔지니어링:
  미리 장애를 만들어 → 취약점 발견 → 수정
  (사전 예방)
```

---

## 2. 카오스 엔지니어링 원칙

```
1. 정상 상태 정의:
   → 측정 가능한 안정 상태 지표 설정
   (예: 초당 성공 결제 수 > 100)

2. 가설 수립:
   → "X 장애가 발생해도 정상 상태를 유지한다"

3. 실제 세계 이벤트:
   → 서버 다운, 네트워크 지연,
      디스크 가득 참, CPU 스파이크

4. 프로덕션에서 실험:
   → 스테이징에서의 결과는 프로덕션과 다를 수 있다

5. 폭발 반경 최소화:
   → 작게 시작, 점진적 확대
   → 즉시 중단 가능한 메커니즘 필수
```

---

## 3. 실험 대상

```
인프라:
  □ Pod/노드 강제 종료
  □ 네트워크 지연/패킷 드롭
  □ 디스크 가득 참
  □ CPU/메모리 과부하

애플리케이션:
  □ 의존성 서비스 다운
  □ DB 연결 끊김
  □ 느린 응답 (레이턴시 주입)
  □ 랜덤 응답 오류

쿠버네티스:
  □ etcd 지연
  □ 노드 드레인
  □ 네임스페이스 네트워크 분리
```

---

## 4. 성숙도 모델

| 단계 | 설명 |
|------|------|
| 1. 수동 게임데이 | 사람이 직접 장애 주입, 반기 1회 |
| 2. 자동화 | CI에서 카오스 테스트 자동 실행 |
| 3. 지속적 카오스 | 프로덕션에서 상시 소규모 실험 |

---

## 5. 실험 전 체크리스트

```
실험 전:
  □ 정상 상태 메트릭 기록
  □ 즉시 중단 버튼 준비
  □ 팀 전체 공지
  □ 비즈니스 영향 낮은 시간대 선택

실험 중:
  □ 메트릭 실시간 모니터링
  □ 폭발 반경 이상 확대 시 즉시 중단

실험 후:
  □ 정상 상태 회복 확인
  □ 결과 문서화
  □ 취약점 발견 시 이슈 등록
```
