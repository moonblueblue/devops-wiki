# On-call 로테이션

> **2026년의 자리**: On-call은 SRE 원칙 중 *가장 먼저 무너지는 축*. Google
> SRE Book *"Being On-Call"*과 Workbook *"On-call"*이 표준. 핵심은 **시프트당
> 페이지 ≤ 2건**, **페어 운영**(Primary + Secondary), **공정 보상**, **사고
> 후 회복 시간 보장**. 2025년 DORA 보고서 65% 엔지니어가 번아웃을 보고 —
> On-call 설계는 *조직 건강 지표*.
>
> 1~5인 환경에서는 6명 표준 로테이션 불가. **공동 On-call**(개발-운영 페어)·
> **부분 커버리지**(영업 외 비페이지)·**자동화 우선**으로 대응.

- **이 글의 자리**: [Incident Response](incident-response.md) 다음. 사고 대응
  시스템의 *사람 측*.
- **선행 지식**: IR 역할 분리, 알람·페이저 도구.

---

## 1. 한 줄 정의

> **On-call**: "*업무 시간 외 알람을 받고 즉시 대응할 책임을 진 사람의
> 시프트.* 페이저·SLA·보상이 결합된 인프라."

### 왜 On-call이 어려운가

| 어려움 | 의미 |
|---|---|
| **수면 단절** | 야간 호출이 다음날·다음주 생산성에 누적 |
| **불확실성** | "오지 않을 호출"을 24h 대기 |
| **번아웃 누적** | 한 분기 쌓이면 *직급 하락*·이직 |
| **공정성 의문** | 누가 더 자주? 어떻게 보상? |
| **지식 단일점** | "그 사람만 아는 시스템" — On-call에 그가 없으면 사고 |

## 2. Google SRE 표준 — 4가지 원칙

| # | 원칙 | 수치 |
|:-:|---|---|
| 1 | **On-call 시간 비중 ≤ 25%** | 근무 시간 대비 |
| 2 | **시프트당 페이지 ≤ 2건** | (수면 시간 제외) |
| 3 | **로테이션 인원 ≥ 6명** | Primary + Secondary 페어 + 충분한 휴식 |
| 4 | **사고 후 회복 시간 보장** | 야간 호출 후 익일 휴무 또는 단축 근무 |

위 4개를 *모두* 만족하는 팀이 SRE의 표준. 하나라도 어긋나면 번아웃 위험.

### 페이지 부하 임계

| 페이지 / 시프트 | 진단 | 처방 |
|---|---|---|
| **0~2** | 건강 | 유지 |
| **3~5** | 경고 | 알람 노이즈 분석 |
| **6+** | 위기 | 즉시 알람 정리, On-call 인원 추가 |
| **야간 1+ / 매주** | 심각 | 시스템 안정화 우선 |

> 페이지가 많으면 *On-call 사람을 늘리지 말고 시스템을 고쳐라.* 페이지
> 수 자체가 *시스템 건강 지표*.

## 3. 로테이션 모델

### 시프트 길이

| 모델 | 길이 | 장점 | 단점 |
|---|---|---|---|
| **주간 (Weekly)** | 7일 | 표준. 명확한 핸드오프 | 한 주 내내 부담 |
| **반주 (Half-week)** | 3.5일 | 부담 분산 | 핸드오프 잦음 |
| **주말 분리** | 평일·주말 별도 | 주말 부담 균등 | 시프트 수 증가 |
| **24h 시프트** | 1일 | 운영 간단 | 매일 반복 — 번아웃 |

> **표준은 주간**. 주말이 매번 같은 사람이면 *주말 분리* 권장.

### 인원 구성

| 모델 | 인원 | 패턴 |
|---|---|---|
| **단일 (Solo)** | 1명 | *금지* — 휴가·병가 시 무방비 |
| **페어 (Primary+Secondary)** | 2명 | 표준. P가 ACK 못하면 S |
| **트리오** | 3명 | 한 명 휴가 시에도 페어 유지 |
| **Follow the Sun** | 9~15명 (지역 3+) | 24h 분산 — 야간 페이지 X |

### Follow the Sun (FTS)

지역 3곳 이상의 팀이 *각자 낮시간 동안만* 페이지를 받음. 야간 호출 자체가
사라진다.

| 조건 | 의미 |
|---|---|
| **최소 9~15명** | 지역당 3~5명 × 3 지역 |
| **타임존 차이 6~10h** | 미국·유럽·아시아 |
| **공통 도구·언어** | Slack·SLO·문서 |
| **지역간 핸드오프** | 12h 단위 인계 |

> **번아웃 67% 감축** 효과 (Uptime Labs 2025 보고). 인원 부족하면 부분
> FTS도 가능 — 야간만 외부 위탁.

### FTS의 함정

| 함정 | 영향 |
|---|---|
| **핸드오프 손실** | 12h마다 컨텍스트 인계 — 진행 중 사고 정보 누락 (예: GitLab 2017 DB 삭제 사고) |
| **문화·언어 차이** | 같은 회사라도 의사소통 비용 |
| **단일 문화권 vs 다국적** | 같은 회사 글로벌 사무소가 외주·자회사보다 핸드오프 안전 |
| **시차 회의 부담** | 24h 운영 → *전 지역 동시* 회의 X — 비동기 강제 |
| **Major Incident 협응** | 다중 리전 사고 시 sub-IC × 지역 + 글로벌 IC 필요 |

## 4. 핸드오프 — 시프트 교대

### 핸드오프 체크리스트

```markdown
# On-call Handoff — YYYY-MM-DD HH:MM

## From: [출근하는 사람]
## To: [받는 사람]

## 미해결 사고
- [ ] INC-1234 — 결제 지연, 모니터링 중
- [ ] INC-1235 — 지난 주 알람 노이즈 조사 진행 중

## 진행 중 변경
- [ ] payment-v2 카나리 진행률 25%
- [ ] DB 마이그레이션 예정 (T+2일)

## 알람 임계 변경
- [ ] X 알람을 Y로 조정 (어제 false positive)

## 알아야 할 외부 요인
- [ ] 결제 게이트웨이 점검 예정 (T+1일 02:00)
- [ ] 마케팅 캠페인 시작 (T+3일)

## 다음 시프트 우선순위
- [ ] INC-1234 1시간 마다 확인
```

### 핸드오프 미팅 (15분)

| 단계 | 내용 |
|---|---|
| 1. 지난 시프트 사고 요약 | 노이즈 / 진짜 |
| 2. 진행 중 사고 인수인계 | 컨텍스트 + 다음 액션 |
| 3. 예정된 변경·외부 요인 | 점검·캠페인·릴리스 |
| 4. 알람 변경·Runbook 업데이트 | 어제 추가·수정 |
| 5. 질의응답 | 받는 사람 우려 |

> **미팅이 없으면 핸드오프가 아니다.** 비동기 문서만으론 컨텍스트 손실.

## 5. 보상 (Compensation)

### 모델 비교

| 모델 | 의미 | 적용 |
|---|---|---|
| **현금 수당** | 시프트당 정액 (예: 주간 $200, 야간 호출 $100/건) | 미국·유럽 흔함 |
| **휴가 환산 (Time off in lieu)** | 호출 시간 × 1.5 ~ 2배 휴가 | 유럽·일부 한국 |
| **연봉 조정** | On-call 책임 = 직급·연봉 반영 | 시니어 SRE |
| **혼합** | 기본 휴가 + 야간 호출 추가 수당 | 균형 |

> Google은 *"휴가 환산 또는 현금"* 중 선택, 연봉의 일정 비율 상한 (예:
> 5%). 보상 없는 On-call은 *암묵적 착취*.

### 보상 구조의 역인센티브

| 모델 | 역인센티브 |
|---|---|
| **시프트 정액** | *노이즈 알람 방치* — 어차피 같은 보상 |
| **호출 시간 가산** | *호출 일부러 늘리기* — 보상 증가 |
| **휴가 환산** | 휴가 사용률 낮으면 의미 없음 — 강제 사용 |
| **혼합** (정액 기본 + 호출 가산) | 양쪽 인센티브 균형 — 권장 |

> 보상 설계는 *행동을 디자인한다*. 단일 모델은 늘 어느 쪽으로든 왜곡.

### 한국 노동법 체크포인트

| 항목 | 의미 |
|---|---|
| **대기시간** | 근기법 §50③ — *사용자 지휘·감독 하의 대기*는 근로시간. 응답 시간 제약 (예: 30분 내 출근)·호출 빈도가 강할수록 근로로 인정 |
| **응답 시간 SLA** | EU CJEU Matzak (2018): 8분 내 응답 의무 = *전체 대기시간이 근로*. 한국도 유사 방향 |
| **호출 응답 시 근로** | 실제 호출에 대응한 시간은 명백히 근로시간 |
| **연장근로 가산** | 주 12시간 초과 시 통상 임금 50% 가산 |
| **야간근로 가산** | 22시~06시 대응 시 50% 가산 |
| **휴일근로 가산** | 주말 대응 시 50% 가산 |

> **법무·HR과 사전 합의 필수**. 위탁·외주로 우회하는 패턴은 위험.
> 응답 시간을 *느슨하게* 설정 (예: 1시간 내) + 명문화된 자유시간 보장
> 으로 *비-근로*에 가까운 구조 설계. 그러나 빈도·강제성이 높으면 근로
> 시간 가산이 안전.

## 6. 회복 시간 (Recovery)

| 트리거 | 회복 권장 |
|---|---|
| **야간 호출 1건** (22시~06시) | 다음날 출근 2시간 지연 또는 재택 |
| **야간 호출 다중** | 다음날 휴무 |
| **주말 호출 4h+** | 다음 평일 휴무 1일 |
| **시프트당 페이지 5+** | 다음 시프트 면제 + Backlog 정비 시간 |

> *회복 보장은 정책 명문화*. 매니저 재량 X — *권리*로.

## 7. On-call 온보딩 — Shadow → Reverse Shadow → Solo

신규 입사자가 곧바로 Primary 페이저를 받으면 사고 확대로 직결. Google
SRE Workbook의 표준 3단계.

| 단계 | 의미 | 기간 |
|---|---|---|
| **1. Shadow** | 기존 On-call이 페이저 받음, 신규는 *옆에서 관찰* | 1~2 시프트 |
| **2. Reverse Shadow** | 신규가 페이저 받음, 기존이 *옆에서 검증* | 1~2 시프트 |
| **3. Solo** | 신규가 단독 — Secondary 백업 보장 | 표준 |

### 온보딩 체크리스트

```markdown
# On-call Onboarding — [Name]

## 시스템 이해
- [ ] 아키텍처 다이어그램 검토
- [ ] CUJ·SLO 목록 숙지
- [ ] 의존성·외부 SaaS 목록

## 도구·접근 권한
- [ ] PagerDuty / Slack 알람 채널 가입
- [ ] 프로덕션 SSH·kubectl 권한
- [ ] DB 읽기 권한 (긴급 진단)
- [ ] Status page 편집 권한

## Runbook 검토
- [ ] Top 10 알람 Runbook 정독
- [ ] 최근 6개월 포스트모템 검토

## 시뮬레이션
- [ ] Wheel of Misfortune 1회 (시뮬레이션)
- [ ] Live drill 1회 (실제 시스템에 가짜 장애)

## 첫 시프트 준비
- [ ] Shadow 1회
- [ ] Reverse Shadow 1회
- [ ] Solo 시 Secondary와 합의
```

## 8. 시뮬레이션 훈련 — Wheel of Misfortune

Google 표준 — 정기 사고 시뮬레이션.

| 종류 | 형태 | 빈도 |
|---|---|---|
| **Wheel of Misfortune** | 회의실 시나리오 — 한 명이 IC 역할, 진행자가 사고 시나리오 제시 | 분기 1회 |
| **Tabletop** | 시나리오 토론 — 30분 | 분기 1회 |
| **Live Drill** | 실제 시스템에 작은 장애 주입 | 반기 1회 |
| **Pager Test** | 알람 발사 자체 검증 | 월 1회 |

> 시뮬레이션은 *훈련*이자 *Runbook 검증*. 시나리오에서 막힌 단계가 다음
> 분기 개선 항목.

## 9. 심리적 안정 — Pager Anxiety

번아웃의 *기술적 원인*이 페이지 횟수라면, *심리적 원인*은 다음 4가지.

| 원인 | 처방 |
|---|---|
| **무엇을 해야 할지 모름** | Runbook 강제, Shadow 온보딩 |
| **실수 시 비난 두려움** | Blameless 문화 ([포스트모템](../postmortem/postmortem.md)) |
| **혼자라는 고립감** | 페어 운영, 즉시 도움 요청 가능 채널 |
| **24h 긴장** | 회복 시간 보장, 휴가 사용률 추적 |

> "*페이저 울릴까봐 잠 못 자는*" 상태가 3일 이상 = *위험 신호*. 매니저가
> 1:1로 점검.

## 10. 번아웃 — 8가지 구조적 대응

업계 통계: 65% 엔지니어가 지난 1년 번아웃 경험. On-call이 주요 원인.

| # | 대응 | 의미 |
|:-:|---|---|
| 1 | **시프트당 페이지 상한** | 2건 초과 시 시스템 안정화 우선 |
| 2 | **알람 정리 분기 1회** | 노이즈 알람 제거 |
| 3 | **Runbook 강제** | 모든 페이지 알람에 Runbook |
| 4 | **사고 후 회복 시간** | 정책 명문화 |
| 5 | **공정 보상** | 휴가 또는 현금, 투명한 규칙 |
| 6 | **로테이션 인원 충분** | 6명 이상 |
| 7 | **Pre-shift 준비 시간** | 시프트 전 30분 — 변경 사항 검토 |
| 8 | **Post-shift 회고** | 시프트 끝나면 30분 회고·기록 |

## 11. 알람 위생 (Alert Hygiene)

### 알람의 3가지 분류

| 등급 | 정의 | 라우팅 |
|---|---|---|
| **Page** | 즉시 사람 개입 필요 | 페이저 |
| **Ticket** | 영업시간 내 처리 | Slack·이메일 |
| **Log** | 기록만 | 대시보드 |

### 모든 페이지 알람의 4가지 필수 속성

| 속성 | 의미 |
|---|---|
| **Actionable** | 사람이 *지금 할 수 있는 것*이 있어야 |
| **Symptom-based** | 원인 X, *증상*에 알람 (5xx 1% 초과 — 기계 1대 죽음 X) |
| **Documented** | Runbook 링크 필수 |
| **Tested** | 분기 1회 발사 시뮬레이션 |

### 알람 정리 분기 회의

```markdown
# Alert Hygiene Review — YYYY-Q?

## 1. 지난 분기 페이지 통계
- 총 호출: 47건
- 노이즈: 18 (38%)
- 진짜 사고: 12 (26%)
- 진짜 + 자동 회복: 17 (36%)

## 2. 노이즈 알람 (제거 또는 임계 조정)
- AlertA: 분기 7번, 모두 자동 회복 → ticket으로 강등
- AlertB: 분기 5번, false positive → 임계 1% → 5%

## 3. 누락 알람
- 지난 사고 INC-1234 — 알람 없이 고객 보고
  → SLI 추가, Burn Rate 알람 도입

## 4. Runbook 누락
- AlertC, AlertD: Runbook 없음 → 다음 분기까지 작성

## 5. 분기 KPI
- 시프트당 페이지 평균: 1.4 (목표 < 2) ✓
- 야간 페이지 / 시프트: 0.3 (목표 < 0.5) ✓
- 노이즈 비율: 38% (목표 < 25%) ✗ → 액션 1, 2
```

## 13. 1~5인 팀의 On-call 현실론

표준 6명 로테이션이 불가능한 환경. *현실적* 대안.

| 인원 | 권장 모델 |
|---|---|
| **1명 (전담)** | *불가능*. 외부 위탁·시스템 단순화 우선 |
| **2명** | 격주 페어, 사고 시 둘 다 깨움 |
| **3명** | 주간 로테이션, 한 명 휴가 시 페어 유지 |
| **4~5명** | 표준 페어 (P+S), 주간 |
| **5+ 인** | 표준 도입 가능 |

### 1~5인 팀 핵심 룰

| 룰 | 의미 |
|---|---|
| **공동 On-call** | 개발 1명 + 운영 1명 페어 — 부하 분산 |
| **부분 커버리지** | 영업 외 페이지는 *고객 영향 한정* |
| **자동화 우선** | 사람보다 자동 복구 (재시작·롤백) |
| **단순화** | 시스템 단순할수록 페이지 적음 |
| **외부 위탁** (선택) | 야간만 SaaS·MSP에 위임 |

### 미니 On-call 정책 — 한 페이지

```markdown
# On-call Policy — 3인 팀

## 로테이션
- 주간 로테이션, 월요일 09:00 핸드오프
- 페어 = Primary + 모두 백업

## 페이지 정책
- 영업시간 (09-18): 모든 SLO 알람
- 영업 외: SEV1·2 (고객 영향 + SLO Burn) 만
- SEV3·4: 다음 영업일 처리

## 회복
- 야간 호출 30분+ → 다음날 오후 출근
- 주말 호출 1h+ → 다음 평일 1일 휴무

## 보상
- 주간 시프트: 휴가 0.5일
- 야간 호출 / 건: 휴가 0.5일 추가

## 알람 위생
- 분기 첫째 주 화요일 14:00 알람 정리 회의
```

## 14. 안티패턴 — On-call 실패

| 안티패턴 | 증상 | 처방 |
|---|---|---|
| **단일 On-call** | 휴가 시 무방비, 번아웃 | 페어 운영 |
| **페이지 무제한** | 시프트당 10+ 호출 | 알람 정리, 시스템 고치기 |
| **Runbook 없는 페이지** | 매번 처음부터 디버깅 | Runbook 강제 |
| **보상 없음** | 이직, 채용 어려움 | 휴가 또는 현금 |
| **회복 시간 무시** | 누적 번아웃 | 정책 명문화 |
| **개발팀 미참여** | SRE에만 부담 | 공동 On-call |
| **야간 페이지 잦음** | 만성 수면 단절 | 시스템 안정화 우선 |
| **노이즈 무시** | 진짜 사고 묻힘 | 분기 알람 정리 |
