# 장애 대응 프로세스

## 1. 전체 흐름

```
탐지 (Detect)
  ↓
트리아지 (Triage) — 심각도 결정
  ↓
대응 (Respond) — 팀 소집, 커뮤니케이션
  ↓
완화 (Mitigate) — 임시 조치
  ↓
복구 (Resolve) — 근본 원인 제거
  ↓
포스트모템 (Postmortem)
```

---

## 2. 역할 분담

| 역할 | 책임 |
|------|------|
| Incident Commander (IC) | 전체 조율, 의사결정 |
| Technical Lead | 기술적 원인 파악, 복구 실행 |
| Comms Lead | 이해관계자 커뮤니케이션 |
| Scribe | 타임라인 기록 |

```
원칙: 한 사람이 두 역할 동시 수행 금지
      IC는 직접 기술 작업 하지 않음
```

---

## 3. 단계별 체크리스트

### 탐지 (0~5분)

```
□ 알림 수신 확인
□ 실제 장애인지 오탐인지 판단
□ 심각도(SEV) 결정
□ IC 지정
□ 인시던트 채널 생성 (#incident-YYYY-MM-DD)
```

### 트리아지 (5~15분)

```
□ 영향 받는 사용자/시스템 파악
□ 관련 팀 호출
□ 외부 의존성(클라우드, 써드파티) 확인
□ 최근 배포/변경 확인
□ 상태 페이지 업데이트 (공개 서비스)
```

### 완화 (15~60분)

```
□ 트래픽 우회 (롤백, 피처 플래그 비활성화)
□ 스케일 아웃
□ DB 읽기 전용 모드 전환
□ CDN 캐시 강제 갱신
□ 회로 차단기(Circuit Breaker) 활성화
```

### 복구

```
□ 근본 원인 확인
□ 수정 배포 (완화와 구별!)
□ 모니터링으로 정상화 확인
□ 인시던트 종료 선언
□ 이해관계자 최종 업데이트
```

---

## 4. 커뮤니케이션 템플릿

```
[초기 알림]
🔴 SEV-1 인시던트 발생
서비스: 결제 API
증상: 결제 실패율 35% (정상 < 0.1%)
영향: 약 XXX 사용자
시작: 14:23 KST
담당: @홍길동 (IC), @김철수 (Tech)
채널: #incident-2026-04-14-payment

[30분 업데이트]
🟡 상태 업데이트 (14:53 KST)
진행: DB 연결 풀 고갈 원인 파악
조치: 연결 수 임시 증가 (100→500)
결과: 실패율 35% → 8% 개선
다음 업데이트: 30분 후

[해결]
✅ 인시던트 해결 (15:40 KST)
원인: DB 연결 풀 고갈
해결: 설정 변경 + 재배포
영향 시간: 1시간 17분
포스트모템: 내일 10:00 예정
```

---

## 5. 롤백 결정 기준

```
즉시 롤백:
  □ 배포 후 5분 내 에러율 급증
  □ 핵심 기능 장애
  □ 데이터 손상 가능성

수정 배포 선택:
  □ 롤백이 더 위험한 경우 (DB 마이그레이션)
  □ 명확한 수정 방법이 있고 배포 시간 < 15분
```
