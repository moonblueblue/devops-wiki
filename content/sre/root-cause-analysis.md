# 5 Whys와 Fishbone Diagram

## 1. 근본 원인 분석 (RCA)

장애의 표면적 원인이 아니라
**왜 그 원인이 발생했는가**를 추적한다.

```
표면 원인: "서버가 다운됐다"
근본 원인: "용량 계획 프로세스가 없었다"
```

---

## 2. 5 Whys

Why를 5번 반복해 근본 원인을 찾는다.

### 예시: 결제 서비스 장애

```
현상: 결제 API 응답 시간 10배 증가

Why 1: 왜 응답이 느려졌나?
  → DB 쿼리 타임아웃 발생

Why 2: 왜 쿼리가 타임아웃됐나?
  → 특정 쿼리가 풀 스캔을 일으킴

Why 3: 왜 풀 스캔이 일어났나?
  → 인덱스가 없는 컬럼으로 조회

Why 4: 왜 인덱스가 없었나?
  → 신규 기능 개발 시 DB 리뷰 없이 배포

Why 5: 왜 DB 리뷰가 없었나?
  → 배포 프로세스에 DB 변경 리뷰 단계 없음

근본 원인:
  배포 프로세스에 DB 스키마·쿼리 리뷰 단계 누락

조치:
  PR 체크리스트에 DB 변경 리뷰 항목 추가
```

## 3. Fishbone Diagram (이시카와 다이어그램)

```
여러 가능한 원인을 동시에 탐색할 때 사용한다.

                  [환경]          [사람]
                    ↘               ↙
[방법] ───────────────────────── → [장애]
                    ↗               ↖
                  [도구]          [재료]
```

### 카테고리 (PEOPLE 프레임워크)

```
Process (프로세스):
  → 절차 누락, 잘못된 순서

Environment (환경):
  → 인프라, 네트워크, 클라우드 환경

Organization (조직):
  → 커뮤니케이션 부재, 책임 불명확

People (사람):
  → 교육 부족, 피로

Legacy (레거시):
  → 오래된 시스템, 기술 부채

Equipment (장비·도구):
  → 모니터링 부재, 도구 미비
```

---

## 4. RCA 진행 방법

```
1. 사실 수집
   → 장애 타임라인 정리
   → 메트릭, 로그, 변경 이력 확인

2. 팀 RCA 세션 (30~60분)
   → 관련자 전원 참석
   → 비난 없는 분위기 유지
   → 화이트보드에 5 Whys 진행

3. 원인 검증
   → 가설이 실제 데이터와 일치하는지 확인
   → 재현 가능한지 확인

4. 조치 수립
   → 각 원인 레벨별 조치
   → 즉시/단기/장기 구분
```

## 5. 오류 유형 분류

| 유형 | 설명 | 대응 방향 |
|------|------|---------|
| 기술적 오류 | 코드 버그, 설정 오류 | 코드 수정, 자동화 테스트 |
| 운영 오류 | 잘못된 절차 수행 | 절차 개선, 자동화 |
| 설계 오류 | 잘못된 아키텍처 | 설계 변경 |
| 외부 의존성 | 써드파티 장애 | 서킷 브레이커, 대안 |
