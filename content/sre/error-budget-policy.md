# 에러 버짓 정책

## 1. 정책의 목적

에러 버짓 정책은 배포 속도와 신뢰성 사이의
균형을 조율하는 공식 합의이다.

```
에러 버짓 많이 남음:
  → 빠른 배포, 실험 가능

에러 버짓 소진 중:
  → 배포 속도 줄이기

에러 버짓 소진:
  → 배포 동결, 신뢰성 개선 집중
```

---

## 2. 정책 문서 구조

```markdown
# 에러 버짓 정책 — payment-api

## 서비스 정보
- 소유팀: 결제 개발팀
- SRE 담당: 플랫폼팀
- SLO: 99.9% (30일 롤링)
- 에러 버짓: 43.8분/월

## 버짓 단계별 조치

### 정상 (> 50% 남음)
- 정상 배포 허용
- 주간 SLO 리뷰

### 주의 (20~50% 남음)
- 배포 전 SRE 검토 필수
- 리스크 높은 변경 다음 달로 연기

### 경고 (10~20% 남음)
- 긴급 수정만 배포 허용
- 일간 SLO 리뷰
- 신뢰성 개선 항목 Sprint 포함

### 동결 (< 10% 남음)
- 신규 기능 배포 금지
- 버그 수정·보안 패치만 예외
- SRE + 개발팀 공동 신뢰성 스프린트

## 버짓 리셋
- 매월 1일 자동 리셋
- 소진 후 다음 리셋까지 동결 유지
```

---

## 3. 예외 처리

```
버짓 없이도 배포 허용:
  □ 심각한 보안 취약점 패치
  □ 법적 규정 준수 변경
  □ 사용자 데이터 손실 방지

예외 승인 프로세스:
  1. 개발팀장 + SRE 리드 승인
  2. 배포 전 롤백 계획 필수
  3. 이후 포스트모템 작성
```

---

## 4. 책임 분배

| 상황 | 개발팀 책임 | SRE 책임 |
|------|-----------|---------|
| 버짓 소진 | 신뢰성 개선 | 도구·플랫폼 지원 |
| 인프라 장애 | 없음 | 복구 주도 |
| 배포 장애 | 롤백·수정 | 온콜 지원 |
| 외부 의존성 | 장애 대응 공동 | 에스컬레이션 |

## 5. 에러 버짓 소비 대시보드

```bash
# Grafana 대시보드 항목
- 현재 에러 버짓 잔량 (%)

- 번-레이트 추이 그래프
- 예상 버짓 소진 날짜
- 최근 배포 이벤트 오버레이
```
