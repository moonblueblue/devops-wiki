# 장애 보고서 작성 요령

## 1. 보고서의 목적

```
목적:
  □ 장애 경위 공식 기록
  □ 원인 분석 및 재발 방지
  □ 조직 전체 학습 자료

원칙:
  비난 없음 (Blameless):
    → 사람이 아닌 시스템과 프로세스에 집중
    → 실수를 드러내기 안전한 문화 필수
```

---

## 2. 보고서 구조

```markdown
# 장애 보고서: [서비스명] [날짜]

## 요약
- 발생 시각: 
- 해결 시각: 
- 총 영향 시간: 
- 심각도: SEV-X
- 영향: 사용자 X명, 요청 X건 실패

## 타임라인
| 시각 | 내용 |
|------|------|
| 14:23 | 알림 발생 |
| 14:28 | IC 지정, 조사 시작 |
| 14:45 | 원인 가설 1 확인 — 불일치 |
| 15:10 | 근본 원인 파악 (DB 연결 풀 고갈) |
| 15:35 | 수정 배포 시작 |
| 15:40 | 정상화 확인, 인시던트 종료 |

## 근본 원인
[5 Whys 분석 결과]

## 영향
- 결제 실패율: 정상 0.1% → 장애 중 35%
- 영향 사용자: 약 12,000명
- 예상 매출 손실: 약 X원

## 조치 사항
| 항목 | 담당자 | 기한 | 상태 |
|------|--------|------|------|
| DB 연결 풀 모니터링 추가 | 김철수 | 04/21 | 진행 중 |
| 연결 풀 고갈 알림 추가 | 이영희 | 04/18 | 완료 |
| 부하 테스트 시나리오 추가 | 박민준 | 04/28 | 예정 |

## 잘한 점 (What Went Well)
- 알림 발생 후 5분 내 IC 지정 완료
- 롤백 결정이 빠르게 이루어짐

## 아쉬운 점 (What Could Be Better)
- 연결 풀 메트릭이 없어 진단 지연
- 커뮤니케이션 채널 늦게 개설
```

## 3. 타임라인 작성 요령

```
좋은 타임라인:
  □ 모든 시각은 KST로 통일
  □ 가설과 확인 결과를 명확히 구분
  □ 오판·실수도 숨기지 않고 기록
  □ 외부 의존성 이슈도 포함

피해야 할 것:
  □ "X가 실수했다" 같은 개인 비난
  □ 사후에 알게 된 정보를 실시간인 것처럼 기술
```

---

## 4. 조치 사항 품질 기준

```
좋은 액션 아이템:
  □ 구체적 (무엇을 어떻게)
  □ 담당자 명확
  □ 기한 명시
  □ 추적 가능 (JIRA/Linear 티켓)

나쁜 예:
  "모니터링 개선" (막연함)
  "시스템 안정화" (측정 불가)

좋은 예:
  "DB 연결 풀 사용률 80% 초과 시
   Slack #alerts 채널에 알림 추가 (담당: 김철수, 04/18)"
```

---

## 5. 보고서 배포

```
내부 배포:
  → 관련 팀 전체 (72시간 이내)
  → 경영진 요약본 별도

외부 배포 (공개 서비스):
  → Status Page 업데이트
  → 큰 장애는 공개 포스트모템 블로그 게시
  (GitHub, Cloudflare 사례 참조)

보관:
  → Wiki/Confluence 장애 아카이브
  → 분기 리뷰 시 활용
```
