# 모니터링 Anti-Pattern

## 1. 알림 피로 (Alert Fatigue)

**증상**: 알림이 너무 많아 실제 장애를 놓친다.

```
Bad:
  → CPU 70% 이상 → 알림
  → CPU 75% 이상 → 알림
  → CPU 80% 이상 → 알림
  결과: 하루 100개 이상 알림, 모두 무시
```

**해결:**

```
Good:
  → CPU 90% 이상 + 5분 지속 → 경고
  → 에러율 1% 초과 → 즉시 알림
  → 비즈니스 임팩트 기준으로 알림 정의
```

| 알림 수준 | 기준 | 응답 |
|---------|------|------|
| Critical | 서비스 다운, 데이터 손실 위험 | 즉시 (24/7) |
| Warning | 성능 저하, 조만간 문제 | 업무 시간 내 |
| Info | 참고용 | 알림 안 함 (대시보드만) |

---

## 2. 가비지 메트릭 수집

**증상**: 실제 쓰지 않는 메트릭을 수천 개 수집한다.

```
Bad:
  → 모든 내부 함수 호출을 메트릭으로 기록
  → 저장 비용 폭증, 쿼리 느려짐
  → 정작 중요한 지표가 무엇인지 모름
```

**해결:**

```
Good:
  → 비즈니스 임팩트가 있는 지표만 수집
  → 새 메트릭 추가 시 "이 지표가 장애 감지에 직접 기여하는가?" 질문
  → 6개월 이상 참조 안 된 메트릭 제거
```

---

## 3. 인프라 지표만 보기

**증상**: CPU, 메모리는 정상인데 사용자는 느리다고 신고한다.

```
Bad:
  → CPU: 30%, 메모리: 50%, 네트워크: 정상
  → "서버는 멀쩡한데 왜 느리지?"
```

**해결:**

```
Good:
  → 인프라 지표 + 애플리케이션 지표 + 비즈니스 지표
  → 응답시간 p95, 에러율, 처리량을 핵심 지표로 사용
  → 사용자 관점의 SLI/SLO 정의
```

---

## 4. 사후 분석 없는 알림

**증상**: 알림이 오면 임시방편으로 처리하고 원인 분석을 안 한다.

```
Bad:
  → 서비스 재시작으로 해결
  → "다시 잘 됩니다" 보고
  → 다음 날 같은 알림 반복
```

**해결:**

```
Good:
  → 모든 장애에 대해 근본 원인 분석 (RCA)
  → 재발 방지 액션 아이템 등록
  → 포스트모템 작성 (SRE 챕터 참고)
```

---

## 5. 로그만 보는 모니터링

**증상**: 대시보드 없이 grep으로 로그 검색만 한다.

```
Bad:
  → 문제 발생 후 로그를 직접 뒤짐
  → 시스템 전반 상태 파악 불가
  → 장애 전조 증상 포착 불가
```

**해결:**

```
Good:
  → 골든 시그널 대시보드 항상 표시
  → 로그는 원인 분석용, 알림은 메트릭 기반
  → 정량적 SLI/SLO로 시스템 건강도 정의
```

---

## 6. 체크리스트 요약

```
□ 알림 수 주기적으로 리뷰 (월 1회)
□ 알림이 울릴 때 행동 가능한(Actionable) 내용인지 확인
□ 인프라 + 앱 + 비즈니스 지표 모두 커버
□ 대시보드를 사람이 실제로 읽는지 확인
□ 장애 후 포스트모템 작성 습관화
□ 사용하지 않는 메트릭/대시보드 정기 정리
```
