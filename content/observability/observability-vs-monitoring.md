# 모니터링 vs 옵저버빌리티

## 1. 모니터링 (Monitoring)

미리 알고 있는 문제가 발생했는지 추적한다.
**"무엇이 잘못됐는가?"**

```
임계값 설정 → 초과 시 알림
  예: CPU 90% 이상 → 경고
  예: 에러율 5% 이상 → 경고
```

특징:
- 사전에 정의된 메트릭과 임계값
- 알려진 장애 패턴에 효과적
- 모르는 문제는 감지 못함

---

## 2. 옵저버빌리티 (Observability)

시스템의 내부 상태를 외부 출력(메트릭, 로그, 트레이스)으로
**이해할 수 있는 능력**.
**"왜 잘못됐는가?"**

```
알 수 없는 문제 발생
  → 메트릭으로 어떤 서비스에서 → 언제
  → 로그로 무슨 에러가 → 어떤 요청에서
  → 트레이스로 어느 단계에서 → 얼마나 걸렸는지
```

특징:
- 사전 정의 없이도 원인 추적 가능
- 복잡한 분산 시스템에서 필수
- 미지의 문제(Unknown Unknown)에 대응

---

## 3. 비교

| 항목 | 모니터링 | 옵저버빌리티 |
|-----|--------|------------|
| 질문 | 무엇이 잘못됐는가 | 왜 잘못됐는가 |
| 방향 | 외부 → 내부 (블랙박스) | 내부 → 외부 (화이트박스) |
| 대상 | 알려진 실패 패턴 | 알 수 없는 실패 패턴 |
| 데이터 | 메트릭, 알림 | 메트릭 + 로그 + 트레이스 |
| 적합 환경 | 단순한 시스템 | 복잡한 분산 시스템 |

---

## 4. 실무에서의 의미

```
모니터링만으로 부족한 상황:

마이크로서비스 A → B → C → D 호출 체인에서
사용자가 느리다고 신고
  → 어떤 서비스가 병목? CPU/메모리는 정상
  → 로그를 일일이 찾아야 함
  → 원인 파악까지 수 시간
```

```
옵저버빌리티가 갖춰진 상황:

분산 트레이스로 A→B→C→D 전체 흐름 1초 만에 시각화
  → C에서 DB 쿼리 2초 지연 즉시 확인
  → 원인 파악 5분
```

---

## 5. 관계

```
옵저버빌리티 ⊃ 모니터링

옵저버빌리티는 모니터링을 포함하는 더 넓은 개념.
모니터링은 옵저버빌리티를 달성하기 위한 수단 중 하나.
```

좋은 옵저버빌리티를 가진 시스템은:
- 운영 중 문제를 빠르게 감지하고
- 근본 원인을 빠르게 파악하며
- 재발 방지를 위한 인사이트를 제공한다
